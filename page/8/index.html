Linux中IPC方式

  1. 管道、流管道、命名管道FiFO

    • 管道：具有两种限制。一是半双工的通信，数据只能单向流动，二是只能在具有亲缘关系的进程间使用。进程的亲缘关系通常是指父子进程关系。
    • 流管道：去除了第一种限制,可以双向传输.
    • 命名管道：除具有管道所具有的功能外，它还允许无亲缘关系进程间的通信。

  2. 信号

    信号是一种比较复杂的通信方式，用于通知接收进程某个事件已经发生。

    主要作为进程间以及同一进程不同线程之间的同步手段。

  3. 信号量

    信号量是一个计数器，可以用来控制多个进程对共享资源的访问。它常作为一种锁机制，防止某进程正在访问共享资源时，其他进程也访问该资源。

    主要作为进程间以及同一进程内不同线程之间的同步手段。

  4. 共享内存

    使得多个进程可以访问同一块内存空间，是最快的可用IPC形式。是针对其他通信机制运行效率较低而设计的。往往与其它通信机制，如信号量结合使用，来达到进程间的同步及互斥。

  5. 消息队列

    消息队列是由消息的链表，存放在内核中并由消息队列标识符标识。消息队列克服了信号传递信息少、管道只能承载无格式字节流以及缓冲区大小受限等缺点。

  6. socket

Java的5种线程状态

发表于 2018-01-13 | 分类于 Java
  1. 新建状态new
  2. 就绪状态runnable
  3. 运行状态running
  4. 阻塞状态blocked
  5. 终止状态dead

阻塞的三种情况

  • 等待阻塞

    运行的线程执行wait()方法，该线程会释放占用的所有资源，JVM会把该线程放入“等待池”中。进入这个状态后，是不能自动唤醒的，必须依靠其他线程调用notify()或notifyAll()方法才能被唤醒，

  • 同步阻塞

    运行的线程在获取对象的同步锁时，若该同步锁被别的线程占用，则JVM会把该线程放入“锁池”中。

  • 其他阻塞

    运行的线程执行sleep()或join()方法，或者发出了I/O请求时，JVM会把该线程置为阻塞状态。当sleep()状态超时、join()等待线程终止或者超时、或者I/O处理完毕时，线程重新转入就绪状态。

java死锁的原因

suspend()挂起以及不指定超时期限的wait()方法的调用

wait和suspend、sleep区别

wait会释放线程占用的锁、而suspend以及sleep不会

RTSP协议详解

发表于 2018-01-13 | 分类于 计算机网络

RTSP协议简介

RTSP是研究如何有效地在IP网络上传输流媒体数据的应用层协议。RTSP对流媒体提供了诸如暂停，快进等控制，而它本身并不传输数据，RTSP的作用相当于流媒体服务器的远程控制。服务器端可以自行选择使用TCP或UDP来传送串流内容。

RTSP的对带宽的要求更低，RTMP实时性更好

特点：

  • 允许同时多个串流需求控制（Multicast），支持多方视频会议
  • rtsp还提供网络负载均衡的功能，减轻服务器压力和网络带宽要求，可以实现流量的重新导向。

RTSP和HTTP区别

  1. 相同点
  • 两者都用纯文本来发送消息，且rtsp协议的语法也和HTTP类似。
  1. 不同点
  • RTSP有状态，HTTP无状态，rtsp的命令总是按照顺序来发送。

RTSP和SIP区别

SIP是基于IP的一个应用层控制协议。

  1. 相同点
  • sip和rtsp都是应用层的控制协议，负责一次通信过程的建立和控制和结束，不负责中间的传输部分。
  • 他们都是基于纯文本的信令协议，支持tcp、udp，支持多方通信。
  • sip和rtsp 都使用sdp协议来传送媒体参数，使用rtp（rtcp）协议来传输媒体流。
  1. 不同点
  • rtsp是专门为流媒体制定的协议，在多个媒体流的时间同步方面比sip强大。
  • rtsp还提供网络负载均衡的功能，减轻服务器压力和网络带宽要求。
  • sip一般用来创建一次音频、视频通话（双向），而rtsp一般用来做视频点播、视频监控等（单向）。从原理上讲，rtsp也可以做双向的视频通话。

RTSP和RTP（rtcp）的关系

rtsp负责建立和控制会话，rtp负责多媒体的传输，rtcp配合rtp做控制和流量统计，他们是合作的关系。

RTSP消息

请求格式

请求消息格式：

方法 URI RTSP版本 CR LF

消息头 CR LF CR LF
消息体 CR LF

响应格式

回应消息格式：

RTSP版本 状态码 解释 CR LF

消息头 CR LF CR LF

消息体 CR LF

RTSP请求方法介绍

  1. OPTION

目的是得到服务器提供的可用方法

  1. DESCRIBE

C向S发起DESCRIBE请求,为了得到会话描述信息(SDP)

  1. SETUP

客户端提醒服务器建立会话,并确定传输模式

  1. PLAY

客户端发送播放请求

  1. TEARDOWN

客户端发起关闭请求

  • 第一步，只要服务器客户端约定好，有哪些方法可用，则option请求可以不要。
  • 第二步，如果我们有其他途径得到媒体初始化描述信息（比如http请求等等），则我们也不需要通过rtsp中的describe请求来完成。

RTSP推流过程

options -> announce(发送一些媒体对象描述信息sdp格式) -> setup（定义实时流的传输模式） -> record(初始化媒体数据记录范围，主要包括开始和结束时间)

RTMP详解

发表于 2018-01-13 | 分类于 计算机网络

RTMP握手过程

握手复杂规则：

  • 客户端要等收到S1之后才能发送C2
  • 客户端要等收到S2之后才能发送其他信息（控制信息和真实音视频等数据）
  • 服务端要等到收到C0之后发送S1
  • 服务端必须等到收到C1之后才能发送S2
  • 服务端必须等到收到C2之后才能发送其他信息（控制信息和真实音视频等数据）

实际上为了在保证握手的身份验证功能的基础上尽量减少通信的次数，一般的发送顺序是这样的：


｜client｜Server ｜

｜－－－C0+C1—->|

｜<－－S0+S1+S2– |

｜－－－C2-－－－> ｜

Message

  • Timestamp（时间戳）：消息的时间戳（但不一定是当前时间，后面会介绍），4个字节
  • Length(长度)：是指Message Payload（消息负载）即音视频等信息的数据的长度，3个字节
  • TypeId(类型Id)：消息的类型Id，1个字节
  • Message Stream ID（消息的流ID）：每个消息的唯一标识，划分成Chunk和还原Chunk为Message的时候都是根据这个ID来辨识是否是同一个消息的Chunk的，4个字节，并且以小端格式存储

Chunk

  1. Basic Header

    包含了chunk stream ID（流通道Id）和chunk type（chunk的类型），chunk stream id一般被简写为CSID，用来唯一标识一个特定的流通道，chunk type决定了后面Message Header的格式。

  2. Message Header（消息的头信息）

    包含了要发送的实际信息（可能是完整的，也可能是一部分）的描述信息。包含前面Message里面提到的4个字段。

chunk size默认为128字节，需要根据实际的带宽情况进行动态调整。

协议控制消息（Protocol Control Message）

在RTMP的chunk流会用一些特殊的值来代表协议的控制消息，它们的Message Stream ID必须为0（代表控制流信息），CSID必须为2，Message Type ID可以为1，2，3，5，6。

  • Set Chunk Size
  • Abort Message
  • Acknowledgement
  • Window Acknowledgement Size
  • Set Peer Bandwidth

注意这是属于RTMPchunk流层的消息，和RTMP中的用户控制消息不同。

RTMP 七种Message

  1. Command Message(命令消息，Message Type ID＝17或20)：表示在客户端盒服务器间传递的在对端执行某些操作的命令消息，如connect表示连接对端，对端如果同意连接的话会记录发送端信息并返回连接成功消息，publish表示开始向对方推流，接受端接到命令后准备好接受对端发送的流信息。
  2. Data Message（数据消息，Message Type ID＝15或18）：传递一些元数据（MetaData，比如视频名，分辨率等等）或者用户自定义的一些消息。
  3. Shared Object Message(共享消息，Message Type ID＝16或19)：表示一个Flash类型的对象，由键值对的集合组成，用于多客户端，多实例时使用。
  4. Audio Message（音频信息，Message Type ID＝8）：音频数据。
  5. Video Message（视频信息，Message Type ID＝9）：视频数据。
  6. Aggregate Message (聚集信息，Message Type ID＝22)：多个RTMP子消息的集合。
  7. User Control Message Events(用户控制消息，Message Type ID=4):告知对方执行该信息中包含的用户控制事件，比如Stream Begin事件告知对方流信息开始传输。和前面提到的协议控制信息（Protocol Control Message）不同，这是在RTMP协议层的，而不是在RTMP chunk流协议层的，这个很容易弄混。该信息在chunk流中发送时，Message Stream ID=0,Chunk Stream Id=2,Message Type Id=4。

View的工作原理

发表于 2018-01-11 | 分类于 Android

View的绘制流程从ViewRoot（对应与ViewRootImpl类）的performTraversals方法开始，经过measure、layout、draw三个过程才能最终将一个View绘制出来。

调用链

performMeasure -> measure -> onMeasure -> measure(View) -> …

performLayout -> layout -> onLayout -> layout(View) -> …

performDraw -> draw -> onDraw -> draw(View) -> …

唯一的小区别是draw过程中onDraw()方法只是绘制当前View，绘制子View的传递是在draw方法中通过dispatchDraw实现的。

支持wrap_content

需要在onMeasure方法中设置specMode == AT_Most时的setMeasuredDimesion()

处理padding

在onDraw方法中通过getPadding()方法得到设置的padding值，进而针对性绘制

自定义View的分类（4种）

继承View重写onDraw方法

主要用于实现一些不规则的效果。

需要自己支持wrap_content以及处理padding

继承ViewGroup实现新的Layout布局

比较复杂，需要处理ViewGroup的测量和布局两个过程，并且同时处理子元素的测量和布局过程。

继承特定的View（比如TextView）

主要用于扩展某种已有的View功能

不需要自己支持wrap_content和padding

继承特定的ViewGroup（比如LinearLayout）

一般用于某种效果可以用几种View组合起来实现时

不需要自己处理ViewGroup的测量和布局过程

线程和进程的区别与联系

发表于 2018-01-10 | 分类于 操作系统
  1. 进程是系统资源分配的基本单位，而线程是cpu调度的基本单位。
  2. 进程有自己独立的地址空间，而线程没有。
  3. 每个线程虽然有一个程序运行的入口、顺序执行序列和程序的出口。但是线程不能独立执行，必须依存在应用程序中，由应用程序提供多个线程执行控制。
  4. 一个程序至少有一个进程,一个进程至少有一个线程.

多线程和多进程的区别

进程有独立的地址空间，一个进程崩溃后，在保护模式下不会对其它进程产生影响==，而线程只是一个进程中的不同执行路径。线程有自己的堆栈和局部变量，但线程之间没有单独的地址空间，==一个线程死掉就等于整个进程死掉，所以多进程的程序要比多线程的程序健壮，但在进程切换时，耗费资源较大，效率要差一些。但对于一些要求同时进行并且又要共享某些变量的并发操作，只能用线程，不能用进程。

Android MVC、MVP、MVVM总结

发表于 2018-01-08 | 分类于 Android

Android MVC

  1. View：一般是xml文件
  2. Model：大多对应本地数据库或者网络数据实体，而且还包括一些简单的数据处理的逻辑。

  3. Controller：activity，主要负责复杂的业务逻辑。

    MVC耦合性相对较高，View可以直接访问model，导致3者之间构成回路

Android MVP

  1. View：一般是activity、fragment（粗粒度的View，View集合）或者是View控件，负责一些简单的业务逻辑。
  2. Presenter沟通View和Model层的桥梁，使得View和Model之间没有耦合，包含复杂的业务逻辑。

  3. Model：主要提供数据的存取功能，数据仓库，包含数据的存取操作。

    View不能直接访问Model，需要通过Presenter发出请求，View与Model不直接通信。

    Android MVVM

    1. View：对应与activity和xml文件
    2. Model：实体模型
    3. ViewModel：负责复杂的业务逻辑

    和MVP很相似，唯一的区别在于View和Model进行双向绑定，两者的变化是同步的。View和Modeld没有直接联系，通过ViewModel进行交互。

单例模式导致的内存泄漏

单例对象中持有需要释放的对象的引用，泄漏场景：activity实现监听器接口并且向单例对象注册监听器之后，并没有解绑监听器。

Java try-catch-finally语句执行的顺序

发表于 2018-01-02 | 分类于 Java

结论

  1. try catch finally 都有返回：最终都会得到finally的返回结果。此时finally中的操作是有效的，影响返回结果。如果finally语句有返回值，则此返回值将会替换掉临时区域原有的值。
  2. try catch 都有返回 finally没有返回：try出现异常，会得到catch的返回结果。finally中的操作，不影响返回结果。
  3. try catch 都有返回 finally没有返回：try没有异常，会得到try的返回结果。 finally中的操作，不影响返回结果。
