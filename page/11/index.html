Android 卡顿的分析思路

发表于 2018-03-15 | 分类于 Android

分析思路

  1. cpu
  2. gpu
  3. 内存

UI渲染的整个过程是由CPU和GPU两个部分协同完成的。

其中, CPU负责UI布局元素的Measure, Layout, Draw等相关运算执行. GPU负责栅格化(rasterization), 将UI元素绘制到屏幕上.

过于复杂的布局(cpu)

  1. UI布局层次太深；
  2. 自定义控件的onDraw中有复杂运算。

以上两点原因会导致CPU的相关运算就可能大于16ms, 导致卡顿。

可以借助Hierarchy Viewer这个工具来帮我们分析布局了. Hierarchy Viewer不仅可以以图形化树状结构的形式展示出UI层级, 还对每个节点给出了三个小圆点, 以指示该元素Measure, Layout, Draw的耗时及性能。

过度绘制(overDraw)(gpu)

Overdraw: 用来描述一个像素在屏幕上多少次被重绘在一帧上。

Android可视化界面分析：”系统设置”–>”开发者选项”–>”调试GPU过度绘制”

造成overdraw的主要原因：

  1. 绘制了多重背景.
  2. 绘制了不可见的UI元素.

UI线程的复杂运算(cpu)

UI线程的复杂运算会造成UI无响应, 当然更多的是造成UI响应停滞, 卡顿.

频繁的GC(内存)

执行GC操作的时候，任何线程的任何操作都会需要暂停，等待GC操作完成之后，其他操作才能够继续运行, 故而如果程序频繁GC, 自然会导致界面卡顿。

造成频繁GC的主要原因：

  • 内存抖动：大量的对象被创建又在短时间内马上被释放。

工程中的原因：
我们在代码的循环中new对象, 或是在onDraw中创建对象等.

计算机中取余和取模运算的区别

发表于 2018-03-05 | 分类于 计算机基础

两个运算的共同点

  1. 求整数商： c = a/b;
  2. 计算模或者余数： r = a - c*b.

两个运算在计算整数商上的区别

  1. 求余运算在取c的值时，向0方向舍入(fix()函数)；
  2. 取模运算在计算c的值时，向无穷小方向舍入(floor()函数)。

在编程语言中常见的表示形式

  1. 取余运算

    rem %
  2. 取模运算

    mod

无论正负，取余运算的结果符号只看前面的操作数符号，和后面操作数符号不相关，至于值的计算和均为正数相同。

关于Dalvik、ART、DEX、ODEX、JIT、AOT、OAT

发表于 2018-03-03 | 分类于 Android

Dalvik和ART

  • DVM基于寄存器，JVM基于栈

  • 执行的字节码文件不一样，DVM执行的是.dex文件，JVM执行的是.class文件。

    1. DVM解释执行的是dex字节码.dex：.java –> .class –> .dex –> .apk
    2. JVM运行的是java字节码.class：.java –> .class –> .jar

  • 运行环境的区别

    1. DVM：允许运行多个虚拟机实例，每一个应用启动都运行一个单独的虚拟机，并且运行在一个独立的进程中
    2. JVM：只能运行一个实例，也就是所有应用都运行在同一个JVM中
  • Dalvik在Android 4.4及以下运行，4.4版本开放了ART的预览版，用户可以自动选择，5.0及5.0之后删除了Dalvik虚拟机，完全改用ART。

ART（Android Runtime）

ART即Android Runtime，是在Dalvik的基础上做了一些优化。在Dalvik下，应用每次运行的时候，字节码都需要通过即时编译器（JIT, just in time）转换为机器码，这会拖慢应用的运行效率，而在ART 环境中，应用在第一次安装的时候，字节码就会预先编译成机器码，使其成为真正的本地应用。这个过程叫做预编译（AOT, Ahead-Of-Time）。这样的话，应用的启动(首次)和执行都会变得更加快速。

Dalvik和ART的区别

Dalvik是运行时解释dex文件，安装比较快，开启应用比较慢，应用占用空间小；ART是安装的时候字节码预编译成机器码存储在本地，执行的时候直接就可以运行的，安装慢，开启应用快，占用空间大。

dex、odex、oat文件

dex文件

dx工具负责将Java字节码.class文件转换为Dalvik字节码.dex，dx工具对Java类文件重新排列，消除在类文件中出现的所有冗余信息，避免虚拟机在初始化时出现反复的文件加载与解析过程。

通常情况下，我们看到的Android应用程序实际上是一个以.apk为后缀名的压缩文件。我们可以通过压缩工具对apk进行解压，解压出来的内容中有一个名为classes.dex的文件。那么我们首次开机的时候系统需要将其从apk中解压出来保存在data/app目录中。

odex文件

如果当前运行在Dalvik虚拟机下，Dalvik会对classes.dex进行一次“翻译”，“翻译”的过程也就是守护进程installd的函数dexopt来对dex字节码进行优化，实际上也就是由dex文件生成odex文件。

  • 生成的odex文件依旧是以.dex为后缀名，保存目录为手机的VM缓存目录data/dalvik-cache

oat文件

如果当前运行于ART模式下， ART同样会在首次进入系统的时候调用dex2oat工具来将dex字节码翻译成本地机器码，保存在data/dalvik-cache下

  • 生成的oat文件依旧是以.dex为后缀名，保存目录为手机的VM缓存目录data/dalvik-cache

Dalvik和ART对dex优化的结果都是保存在一个相同名称的odex文件中

  • oat文件是一个自定义的elf（可执行的二进制文件）文件，里面包含的都是本地机器指令

JIT、AOT

JIT(Just In Time)

  1. JIT在2.2版本时提出
  2. DVM负责解释dex文件为机器码==，如果我们不做处理的话，每次执行代码，都需要Dalvik将java代码由==解释器(Interpreter)将每个java指令转译为微处理器指令，并根据转译后的指令先后次序依序执行，一条java指令可能对应多条微处理器指令，这样效率不高。
  3. Google在2.2版本添加了JIT编译器，当App运行时，每当遇到一个新类，JIT编译器就会对这个类进行编译，经过编译后的代码，会被优化成相当精简的原生型指令码（即native code），这样在下次执行到相同逻辑的时候，速度就会更快。
  4. JIT不对所有dex代码进行编译，而是只编译执行次数较多的dex为本地机器码。

AOT（Ahead Of Time）

  1. JIT是运行时编译，这样可以对执行次数频繁的dex代码进行编译和优化，减少以后使用时的翻译时间，虽然可以加快Dalvik运行速度，但是还是有弊病，那就是将dex翻译为本地机器码也要占用时间。
  2. 在ART 环境中，应用在第一次安装的时候，字节码就会预先编译成机器码。之后打开App的时候，不需要额外的翻译工作，直接使用本地机器码运行，因此运行速度提高。

JIT和AOT对比

  • JIT代表运行时编译策略，也可以理解成一种运行时编译器，是为了加快Dalvik虚拟机解释dex速度提出的一种技术方案，来缓存频繁使用的本地机器码
  • AOT可以理解运行前编译策略，ART虚拟机的主要特征就是AOT

Android N（7.0）以上新的特性

  • 在AOT的编译方式基础上引入JIT，对于经常用的method用AOT方式，对于不经常用的method等用JIT，并且对于这些常用或者不常用维护一个profile。
  • 在设备空闲或者充电的情况做预编译，可以节约时间和省电。

TCP和UDP协议详解

发表于 2018-02-11 | 分类于 计算机网络

TCP报文结构

TCP报文结构详解

三次握手和四次挥手

注意ack和seq的变化，syn同步报文段占用一个序号（相当于一个字节），而fin终止报文段由于没有数据，所以不占用序列号，而且最后的ack报文也不传输数据，所以不占用序列号。

Q: 三次握手建立连接时，发送方再次发送确认的必要性？

A: 主 要是为了防止已失效的连接请求报文段突然又传到了B,因而产生错误。假定出现一种异常情况，即A发出的第一个连接请求报文段并没有丢失，而是在某些网络结 点长时间滞留了，一直延迟到连接释放以后的某个时间才到达B，本来这是一个早已失效的报文段。但B收到此失效的连接请求报文段后，就误认为是A又发出一次 新的连接请求，于是就向A发出确认报文段，同意建立连接。假定不采用三次握手，那么只要B发出确认，新的连接就建立了，这样一直等待A发来数据，B的许多 资源就这样白白浪费了。

Q: 四次挥手释放连接时，等待2MSL的意义？

A:

  1. 为了保证A发送的最有一个ACK报文段能够到达B。这个ACK报文段有可能丢失，因而使处在LAST-ACK状态的B收不到对已发送的FIN和ACK 报文段的确认。B会超时重传这个FIN和ACK报文段，而A就能在2MSL时间内收到这个重传的ACK+FIN报文段。接着A重传一次确认。
  2. 他还可以防止已失效的报文段。客户端在发送最后一个ACK之后，再经过经过2MSL，就可以使本链接持续时间内所产生的所有报文段都从网络中消失。从而保证如果马上建立下一次连接，上一次失效的报文不会干扰此次通信过程。

Q: 为什么是2MSL？

A:

  1. 服务器fin报文超时重传时间一般为1个MSL，然后再考虑到fin报文传递到的客户端的时间最长为1个MSL

UDP

udp支持广播和组播，支持一对一、一对多、多对一、多对多的通信，而tcp不支持广播和组播

UDP报文结构

  1. 源端口：源端口号，在需要对方回信时选用，不需要时可全 0.
  2. 目的端口：目的端口号，在终点交付报文时必须要使用到。
  3. 长度：UDP 用户数据报的长度，在只有首部的情况，其最小值是 8 。
  4. 检验和：检测 UDP 用户数据报在传输中是否有错，有错就丢弃。

UDP校验和的计算方法

现在数据报之前加入12个字节的伪首部，先令2个字节的校验和全为0，然后利用二进制反码求和机制计算出最后的校验和。

伪首部并不是数据报真正的首部，是为了计算校验和而临时添加在数据报前面的，在真正传输的时候并不会把伪首部一并发送。
